# Единый Стандарт Документации (Gold Standard v7.0)

📂 PROJECT_ROOT/
├── 📄 CLAUDE.md                   # 🟢 Точка входа. `<instructions>` + архитектура + anti-patterns.
├── 📂 docs/                        # Документация (читается, не выполняется). Индексируется build-index.sh.
│   │                                Haiku-роутер фильтрует по симптомам — маркетинг не подтянется при отладке кода.

│   │   ├── 📄 fact-*.md           #     Как работает (механика, API, платформа, поведение фич).
│   │   └── 📄 fix-*.md            #     Решения проблем (баги, шрамы, костыли).
│   ├── 📂 methodology/            # Переносимая философия предметной области (UI, тесты как философия).
│   │   └── 📄 design.md           #     UX-философия, визуальный язык, анти-паттерны дизайна.
│   ├── 📂 marketing/              # Продуктовые артефакты для внешнего использования (имя свободное: product/, vision/).
│   │   ├── 📄 _intro.md           #     Карта папки (не индексируется).
│   │   └── 📄 *.md                #     Маркетинг, SEO, юридика, токены, философия продукта, лендинг-тексты.
│   ├── 📂 meta/                   # Ситуативные правила пользователя (не AI): рабочие процессы, шпаргалки.
│   │   └── 📄 *.md                #     Kanban, отчёты, коммуникация — индексируется но релевантно редко.

│   └── 📂 infrastructure/         # 🟢 Ops-инструкции (опционально, можно как fact-ops-*.md).
└── 📂 tests/ scenarios/ scripts/  # Исполняемый код тестов/сценариев. НЕ документация. Где исполняется — там и живёт.

│       └── 📄 <project>.md        #     Подгружается когда задача про конкретный проект.
 

### Ключевые сущности

**`CLAUDE.md`** guard rails + обзор, не каталог knowledge-файлов
    - *`<instructions>` блок* — процессные правила, активные на КАЖДЫЙ промпт: исследуй данные перед кодом, RED TEAM, не защищай при сомнениях
    - *обзор проекта* (стек, архитектурные решения в 1-3 строки) — то что нужно на каждом промпте
    - *anti-patterns* — технические запреты с привязкой к коду: "не используй execSync", "не подписывайся на Map целиком в Zustand"
    - *методологии* — ссылки на `docs/methodology/` и `auto/methodology.md` (читаются по запросу, не всегда)
    - *ссылки на knowledge/ допустимы* только для инфраструктурных инструкций (CI/CD, деплой, SSH) — то что не ищется по симптомам
    - *не должен содержать* каталог ссылок на knowledge-файлы — для этого есть семантический поиск

**`methodology/`** переносимые принципы **построения продукта** — индексируется + ручная активация
    - содержит философию **предметной области**: UX-философия, дизайн-принципы, паттерны взаимодействия. То **что** мы строим, не то **как** мы работаем
    - `design.md` — *"Power Users → 1 клик"*, *"плоская гамма без градиентов на продажных страницах"*, *"выделение через форму, не через цвет"* — работает и в hh-tool, и в custom-terminal, и в будущем проекте
    - индексируется build-index.sh — Haiku-роутер подтянет автоматически когда симптом про дизайн/UX, можно и явно вызвать (*"следуй правилам дизайна"*)
    - растёт с каждой сессией: правка пользователя → *"запиши в правила"* → Claude аппендит
    - **граница с `product/`**: methodology = принцип (*"не называть конкурентов в сравнительных блоках"*), product = артефакт (*"на лендинге hh-tool убран кейс 'Хаос'"*). Тест: *"применю ли я это правило в лендинге другого проекта?"*. Да → methodology. Нет → product
    - **граница с `~/.claude/skills/`**: решается по **широте темы**, не только «продукт vs процесс». Общая многопроектная тема процесса (как подключаться через Electron MCP, как гонять web dev-сервер + browser MCP, как делать backend curl-запросы) → **skill** `~/.claude/skills/<тема>-e2e-testing/`. Проектно-специфичный flow той же темы → либо `references/<project>.md` внутри skill'а (если тема в skill'е уже есть), либо `methodology/` проекта (если flow уникален проекту). Тест ширины: *«эта тема встречается в 2+ проектах?»*. Да → skill (общее) + references (специфика). Нет, тема уникальна одному проекту → methodology/ проекта
        - *пример общей темы:* `electron-e2e-testing`, `web-app-e2e-testing` — Electron/Web тестируется во многих проектах, общая методология в skill, per-project quirks в `references/custom-terminal.md` / `references/uav.md`
        - *пример уникальной темы:* iOS-тестирование используется в ОДНОМ проекте (habit-tracker) — skill заводить рано, flow живёт в `methodology/диагностика-apple.md` проекта. Когда появится второй iOS-проект — тогда выносить в skill
        - *в methodology допустимы flow-как-работает-нейронка*: `диагностика-<домен>.md` (как диагностировать Apple-tooling), `массовое-заполнение-данных.md` (как наполнить БД тестовыми данными), hot-reload расширения — если тема НЕ дозрела до общей. Это не «философия дизайна», но это **проектный flow для нейронки**, и место ему в methodology, не в knowledge (knowledge = механика как работает, methodology = как с этим работать)
    - принцип размещения: правило ВСЕГДА на каждый промпт → `<instructions>` в CLAUDE.md. Домен переносимый между проектами (дизайн, UX-философия) → methodology/. Тема процесса в 2+ проектах → глобальный skill + references. Тема процесса уникальна одному проекту → methodology/ проекта. Специфично для этого продукта (артефакты) → product/ или knowledge/

**methodology = flow-промпт для нейронки-проверяльщика, не нагромождение сценариев.** Файл методологии физически **скармливается нейронке** которая проверяет реализованную фичу или строит новую. Поэтому он пишется как **связное руководство на мета-уровне** («как думать про этот класс задач»), а не как свалка несвязанных кейсов — для разрозненных багов есть `knowledge/fix-*` (там каждый шрам отдельной записью по симптому)
    - **мета-уровень + техописание одновременно**: методология даёт принцип («тестируем поведение, не реализацию»; «каждая категория ошибки = отдельный тест»), но НЕ забывает техническую конкретику, потому что нейронка по этому файлу будет действовать — нужны реальные команды, селекторы, пороги. Чистая абстракция без технических зацепок бесполезна для исполнителя, голый список команд без принципа — не переносится на новый случай
    - **конкретные примеры — в `fix-*`, обобщение — в methodology**: если в methodology накопились конкретные баги один за другим («вот этот popup обрезался», «вот этот dropdown не закрылся») — это сигнал что материал просится в `knowledge/fix-*` (каждый по своему симптому), а в methodology остаётся **принцип** который их объединяет со ссылкой вниз. methodology отвечает «как подходить», fix отвечает «вот конкретный случай и как чинили»
    - **тест на нагромождение**: если из methodology-файла убрать половину секций — он всё ещё учит «как думать про класс»? Да → секции были иллюстрациями принципа, ок. Нет, файл рассыпался в набор несвязанных кейсов → это замаскированный каталог, кейсы расфасовать в `fix-*` по симптомам, в methodology оставить мета-руководство

    **высказывание пользователя = триггер для methodology, не только для knowledge.** Любое утверждение в сессии, которое формулирует *правило* (не описывает локальный фикс), — кандидат в methodology. Формы высказывания, которые надо ловить:
    - *самосформулированный принцип*: *«по философии всё детерминировано»*, *«один клик = одно изменение»*, *«hover не должен предсказывать клик»* — юзер сам обобщил, остаётся записать
    - *контраст «не X, а Y»*: *«это не модалка, а dropdown»*, *«не hover, только клик»*, *«не push, inline»* — явная граница между двумя ментальными моделями
    - *обоснование через категорию*: *«потому что это персистентное состояние»*, *«это выпадающее меню как Rewind»* — ссылка на тип/категорию = правило для всех элементов этой категории
    - *повторение однотипной правки*: *«везде «загрузка» переименуй в «скачиваем аудио»»* — за конкретной правкой стоит правило именования
    - *отвержение формулировки*: *«4 ошибки — непонятно почему»* — нарушение принципа, который надо сформулировать (читаемость, явность)
    - *гипотетический user journey*: *«а если он сначала X, потом Y?»* — путь, не вписывающийся в текущую модель = принцип о гибкости путей

    Разница записей:
    - knowledge: *"в JobProgress.jsx лейбл `downloading` для voice → `скачиваем аудио`"* — описывает текущее состояние кода.
    - methodology: *"stage labels = глагол + объект"* — формулирует **правило** которое работает в любом UI.

    Оба могут существовать одновременно, но methodology обязательна — иначе следующее правило в новом UI напишется тем же ошибочным способом, потому что урок не был обобщён. Тон высказывания (спокойное/эмоциональное) значения не имеет — фильтровать по содержанию: есть в высказывании сформулированное правило или нет.

**`product/`** — продуктовые артефакты для внешнего использования
    - жанры файлов: маркетинг, SEO, юридика, токены/экономика, философия продукта *"зачем это нужно"*, позиционирование, лендинг-тексты, стратегия рекламы, обзоры конкурентов
    - отвечает не на *"как работает код"*, а на *"зачем продукт существует и как мы о нём говорим миру"*
    - примеры имён файлов: `telegram-ads.md`, `seo.md`, `юридика.md`, `ai-токены.md`, `философия-продукта.md`, `обзор.md`, `техническая-справка.md` (для лендинга, не для AI)
    - имя папки свободное — может быть `marketing/` в одном проекте, `vision/` в другом, `product/` в третьем. Содержание определяет назначение, не имя
    - активируется пользователем (*"напиши лендинг"*, *"подготовь юридику"*, *"обнови seo"*) или подтягивается через `docs_search` когда симптом запроса близок (*"какие у нас цены подписки"*, *"что писать в оферте про ТГ"*)
    - индексируется build-index.sh — Haiku-роутер сам решает когда файл релевантен. При отладке кода маркетинговые тексты не подтянутся (роутер фильтрует по симптомам), при вопросе про продукт/юридику — подтянутся
    - отличие от `knowledge/fact-*`: `fact-*` описывает *механику* биллинга (формула, рефанды, race conditions), `marketing/ai-токены.md` описывает *продуктовые принципы* (почему переходим на подписку, курс фиксирован навсегда, экономика для лендинга). Мотивация одного решения может жить в обоих файлах в разных формулировках — для AI-агента в `fact-*`, для человека в `marketing/`
    - `_intro.md` внутри папки — карта файлов (одна строка на файл), помогает Claude и человеку ориентироваться без открытия каждого

**`meta/`** ситуативные правила и инструкции — вызываются вручную когда нужны
    - файлы с узкой областью применения: flow конкретного рабочего процесса, шпаргалка для конкретной ситуации
    - примеры: *"как распределять задачи на Kanban-доске YouGile"*, *"flow отчёта по задаче работодателю"*, *"правила коммуникации с тестировщицей"*, *"шаблон AI-промптов для проекта"*
    - отличие от `methodology/`: methodology — это **правила для AI-агента** (как дизайнить, как тестировать). meta — это **правила для пользователя** (как общаться, как организовать работу), которые пользователь может попросить AI помочь применить
    - отличие от `marketing/`: marketing — продукт наружу (SEO, лендинг, юридика). meta — рабочие процессы внутрь (как я работаю, с кем, по каким правилам)
    - индексируется build-index.sh, но симптомы у этих файлов очень специфичные (*"как заполнить YouGile карточку"*) — Haiku-роутер их не подтянет случайно при коде/маркетинге
    - имя папки может быть `meta/`, `workflows/`, `playbooks/` — содержание важнее имени

**`knowledge/`** всё знание проекта в одной плоской папке. Два типа файлов:

  **`fact-*`** — как подсистема работает
    - поведение фич (user flow, edge cases, code map)
    - внешние факты (API лимиты, платформенные ограничения)
    - *"Timeline показывает Viewport Dash при скролле с дебаунсом 100ms"* → fact-timeline.md
    - *"Chromium ограничивает 16 WebGL контекстов"* → fact-terminal-rendering.md
 
  **`fix-*`** — что пошло не так и как починили
    - баги, race conditions, workarounds, отброшенные подходы
    - записывать если >3 итераций отладки И решение контринтуитивно (не выводится из знания факта)
    - если решение очевидно из факта о системе — записать как секцию в fact-файле, не создавать отдельный fix-
    - *"WebGL крашит драйвер: пробовали dispose, не помогло → Canvas"* → fix- (3 подхода, решение контринтуитивно)
    - *"Sentinel Hub возвращает дубликаты дат с разным Processing Baseline"* → секция в fact- (решение — дедупликация — очевидно из факта)
    - НЕ fix: простое расширение функционала. *"Добавить кнопку"* — это fact-, не fix-

**Naming:** префикс отражает ТИП контента (fact/fix), не домен. Тема — в самом имени.

**`~/.claude/skills/<skill-name>/SKILL.md`** — глобальные skills, переносимые **процедуры работы AI** (не часть `docs/` проекта)
    - содержит **как работать**: процедуры отладки, тестирования, взаимодействия с инструментами. То **как** AI делает, не то **что** строит
    - примеры: `browser-mcp-debug` (что делать когда browser-MCP молчит — sanity-check через `list_tabs`/`curl`), `electron-e2e-testing` (правила E2E-тестов через `mcp__electron__*`)
    - формат: папка `<name>/` с обязательным `SKILL.md` + frontmatter `name` / `description` / `version`. Доп. файлы в `references/<topic>.md` — подгружаются по теме (per-project quirks, специфические таблицы команд)
    - активируется **не по симптому продукта**, а **по типу ситуации**: «получил No response from frame» → browser-mcp-debug, «напиши E2E для drag» → electron-e2e-testing. Frontmatter `description` решает когда AI подтягивает skill
    - **граница с `methodology/`**: methodology = философия предметной области (что строим), skills = процедура работы (как делаем). Тест из предыдущего блока: *«про продукт или про процесс?»*. Дизайн UI → methodology. Отладка/тестирование/инструменты → skills
    - **границы переносимости**: skill **должен быть переносим между проектами**. CDP-порт для одного приложения, project-specific IPC cookbook — это **не** skill, это `references/<project>.md` внутри skill'а (per-project ответвление общей процедуры)
    - не индексируется build-index.sh — skill активируется механизмом `~/.claude/skills/`, не через `docs_search`. Каталог skills остаётся компактным, чтобы все frontmatter'ы помещались в системный промпт

**Тесты, сценарии, исполняемый код — НЕ документация.** Это отдельная ось.
    - Все три категории внутри `docs/` (`knowledge/` / `methodology/` / `product/` / `meta/`) — это **документация**, читаемый markdown
    - Тест-сценарии, scripts, executable artifacts — это **код**, живёт там где исполняется (`tests/`, `scenarios/`, harness-репозиторий)
    - Не складывать сценарии в `docs/knowledge/` «потому что они в markdown». `docs_search` их искать не должен — это спам в результатах поиска
    - Пример: централизованный test harness `~/Desktop/electron-mcp/` хранит **код сценариев** в `scenarios/<project>/`. **Skill** `electron-e2e-testing` хранит **методологию написания** этих сценариев. Per-project quirks (CDP-порты, IPC cookbooks) — в `references/<project>.md` внутри skill'а

    
### Ключевые правила

**место файла определяется паттерном использования, не логической категорией** перед созданием папки или выбором куда положить файл — ответь на два вопроса
    - *как часто этот файл редактируется?* Часто (каждую неделю: маркетинг, SEO, стратегия) → нужна выделенная папка с `_intro.md`, структура, семантические имена файлов. Редко или никогда (правило общения с работодателем, flow отчёта) → `meta/`, плоский набор, не нужна ни индексация, ни иерархия
    - *нужно ли AI автоматически подгружать это при разработке?* Да (баги, механика, шрамы) → `knowledge/`, основная база. Нет (маркетинг, рабочие процессы, шпаргалки) → `marketing/` или `meta/`. Все папки индексируются `build-index.sh`, но Haiku-роутер фильтрует по симптомам — маркетинг не подтянется при отладке кода, knowledge не подтянется при вопросе про юридику
    - группировка по логическому домену (*"взаимодействие в команде"*, *"юридика"*, *"Agile"*) — вторична. Файл про Kanban-доску и файл про отчёт работодателю оба относятся к "рабочим процессам", но если каждый из них вызывается раз в месяц и не редактируется — не нужна папка `рабочие-процессы/`, достаточно `meta/kanban-flow.md` и `meta/отчёт-работодателю.md` рядом
    - *исключение:* если файлов одного домена >5 и все часто редактируются — выделять в папку. Пример: `marketing/` (7 файлов, все активные). Антипример: 2 файла про Agile — не папка, а два файла в `meta/`

**дедупликация** одно знание — одно место
    - CLAUDE.md содержит guard rails и обзор, knowledge/ содержит детали — не дублировать между ними
    - knowledge-файлы могут ссылаться друг на друга для связанных тем

**обнаруживаемость через семантический поиск** knowledge-файлы не требуют ручных ссылок из CLAUDE.md
    - `.semantic-index.json` + MCP `docs_search` находят файлы по тегам и симптомам автоматически
    - качество обнаружения зависит от симптомов в файле, а не от наличия ссылки

**объединение по темам** не создавать кучу мелких файлов, объединять по теме с секциями
    - оптимально 100–300 строк на файл. Больше 300 — тема слишком широкая, разбить
    - если файл покрывает 5+ слабо связанных подсистем с разными симптомами — разбить. Монолитные файлы получают размытые теги при семантической индексации

**AI-to-AI** документация для AI-агента с полным доступом к коду
    - записывать только невидимый контекст: мотивацию, отброшенные подходы, неочевидные связи
    - *"добавили библиотеку X"* — AI найдёт в package.json сам
    - *"Пробовали requestAnimationFrame — мерцание, только double-dispatch решил"* — это сигнал

**нейминг файлов = ассоциация с первого взгляда** имя должно отражать содержание так, чтобы юзер при беглом взгляде на дерево папок понимал что внутри без открытия
    - **кириллица через дефис по умолчанию**: `тарифы-подписок.md`, `дизайн-лендинга.md`, `связи-фич.md`. Юзер думает на русском, имена должны опознаваться без перевода в голове. Никаких `community-pinned-posts.md` / `guidelines.md` / `overseer-prompt.md` — английские словосочетания юзер не считывает с первого взгляда
    - **английский только для устоявшихся коротких техтерминов**: `fact`, `fix`, `bug`, `auto`, `tab`, `prompt`, `docs`, `seo` — это маркеры жанра/слоя, читаются как русские. `react`, `tools`, `notes`, `chats`, `community` — слишком общие, запрещены как имена файлов (могут быть **именами папок** — `methodology/`, `tools/`, `community/`)
    - **не больше одного англ-слова в имени**: `bug-цепочки.md` ✓, `cross-bugs.md` ✗, `feature-checklist.md` ✗. Если англ-слов два — переименовать одно на русский
    - **суффикс `.prompt.md` обязателен** для файлов которые копируются целиком в нейронку как промпт. Без суффикса промпт неотличим от обычной документации
    - **запрещены generic-имена** уровня всего проекта: `guidelines.md`, `notes.md`, `tools.md`, `community.md`, `react.md`, `utils.md`, `helpers.md`, `common.md`. Они ничего не говорят о содержании. Допустимы только: `CLAUDE.md` (зарезервированное имя точки входа), `README.md` (стандарт), `_intro.md` (карта папки)
    - **тест на nейминг**: посмотри на файл в дереве 0.5 сек, отведи взгляд. Можешь сказать что внутри? Нет → имя плохое
    - примеры правильного: `тарифы-подписок.md`, `ai-токены.md`, `сессии-в-заметки.prompt.md`, `bug-цепочки.md`, `fact-credit-billing.md`, `fix-react-ловушки.md`, `обновление-файла.prompt.md`
    - примеры неправильного: `guidelines.md` (что за guidelines?), `community-pinned-posts.md` (английская фраза, не считывается), `react.md` (про что в react?), `overseer-prompt.md` (англ слово + неясно про что), `feature-checklist.md` (генерик + два англ)

**жанры внутри `methodology/`** — мета-файлы для работы с проектом, не обязательно переносимые между проектами
    - **продуктовая методология** (корень `переносимый-дизайн`) — UX-философия + UI implementation patterns. Что строим и как реализуем UI-фичи. Активируется по симптому про дизайн/UX. Один тематический файл `переносимый-дизайн.md`, не плодить узкие подфайлы типа `ui-patterns.md` / `dropdowns.md` / `tooltips.md` — Claude не знает по какому имени искать, лучше большой файл с симптомом-named секциями внутри. Домен-суффикс при необходимости: `переносимый-дизайн-mobile.md` (мобильные клиенты). **Имена собираются из корня + домена** (см. `найминг-файлов.md::корень-сущность + домен-суффикс`), голые `design.md` / `design-bugs.md` в новых файлах не вводить
    - **сценарии багов** (корень `сценарии-багов`) — **любой каталог багов это `сценарии-багов`**, домен уточняет каких:
        - `сценарии-багов.md` (без домена) — кросс-фичевые связи: изменение фичи A ломает фичу B. Граф «трогаешь X — проверь Y и Z». Структура: `### Узел-фича` → «связан с / при изменении проверить / симптом рассинхрона» → ссылка на `fact-<тема>.md`. Активируется **обязательным шагом в `<instructions>` CLAUDE.md** перед изменением фичи. Заводится в проектах с взаимозависимыми фичами (uav, hh-tool, custom-terminal)
        - `сценарии-багов-design.md` — грабли вёрстки одного UI-компонента: popup-обрезка у края, hover-bridge закрытие, click-outside, z-index, focus-ловушки, Portal+overflow. «Что ломается при сборке интерфейса». Читается ПЕРЕД проектированием UI-элемента. Бывшее `design-bugs.md`
        - `сценарии-багов-стек.md` — технические баги стека по подсистемам (RTK Query, Bull, CSS, маппинг данных). Бывшее `типовые-баги.md`
        - все три — один корень `сценарии-багов`, разные домены. Claude видит семью, не россыпь. Поиск предсказуем: «баги дизайна» → `-design`, «баги стека» → `-стек`, «связи фич» → без домена
    - **сценарии использования** (`сценарии-использования.md`) — пошаговые цепочки `«юзер в состоянии X делает Y → ожидает Z»`, проходящие через несколько фич. Не привязаны к одной фиче — описывают **путь юзера через систему**. Активируются явно («проверь следование сценарию X», «не сломали ли мы основной флоу») или через `docs_search` по запросам про user-journey. Один файл на проект, внутри секции `## Сценарий: <имя>` с подсекциями `### Основной путь` / `### Edge: <условие>`. Заводится когда проект имеет нетривиальные user-flow через состояния системы (background/foreground, killed/running, locked/unlocked, online/offline) — то есть когда последовательность действий важнее статики UI. Не нужен в изолированных тулзах с одной точкой входа
    - **корень `сценарии-багов` vs `knowledge/fix-*` не путать**: `сценарии-багов-*.md` (любой домен) — каталог/карта на мета-уровне, «класс багов как думать»; `knowledge/fix-*.md` — конкретный шрам конкретной подсистемы (как чинили, что пробовали). Каталог отсылает к fix за конкретикой. Плюс `сценарии-использования.md` — успешный путь юзера, не баг вообще
    - **что НЕ methodology**: глобальные правила общения юзер↔Claude (тон, формат ответа, причинно-следственная цепочка) — живут в `~/.claude/CLAUDE.md` глобально, инжектятся через GT Editor deploy. Не дублировать по проектам

**user-flow сценарий = свой жанр, не подсекция в fact-файле** распространённая ошибка — записывать пошаговый путь юзера секцией внутри `fact-<подсистема>.md` фичи. На практике это ломает обнаружение: сценарий проходит через 2-3 подсистемы, и в каком из 3 fact-файлов искать его — непонятно. Claude через `docs_search` по запросу «проверь основной флоу записи» не найдёт сценарий, потому что fact-файлы индексируются по симптомам своих подсистем
    - **признак что нужен сценарий**: путь юзера включает переходы между состояниями системы (background↔foreground, killed→cold-launch, locked→unlocked), несколько фич участвуют последовательно, edge-states меняют ход событий. *«killed app → тап Shortcut на залоченном экране → видит pop-out → разблокирует → текст в буфере»* — пять состояний, две фичи (Shortcut intent + Live Activity + clipboard sync), один путь. В fact ни одной подсистемы целиком не помещается, нужен отдельный жанр
    - **признак что сценарий НЕ нужен**: «нажми кнопку → откроется панель» — это UX-реализация (статика), в `fact-<фичи>.md` секцией про user-flow. Сценарий нужен когда есть **цепочка состояний** длиной 3+ шагов с участием системных переходов, не просто описание клика
    - **структура сценария**: заголовок `## Сценарий: <имя>` → одна-две строки контекста (когда юзер этот путь проходит) → `### Основной путь` пронумерованный список шагов с указанием состояния системы на каждом шаге → `### Edge: <условие>` дополнительные секции для прерываний (звонок во время записи, рестарт app, потеря сети) → ссылки на `fact-*` файлы участвующих подсистем в конце
    - **не путать с `сценарии-багов.md`**: `сценарии-использования.md` — успешный путь юзера во времени (флоу работает); `сценарии-багов.md` — карта где фичи ломают друг друга (что проверить при изменении). Разные оси: один про user-journey, другой про регрессии

**стек-ловушки (React/Zustand/Prisma/Node) расфасовываются по симптомам, не в сборный файл**
    - типичная ошибка: создать файл `fix-react-ловушки.md` / `fix-prisma-gotchas.md` / `common-bugs.md` со списком 10 шрамов разных тем. Logically группировка по стеку, на практике — Claude вызывает `docs_search` с запросами «как реализовать X», «session ID дрожит между значениями», «кнопка в Portal не работает» — Haiku-роутер ищет по симптомам, не по «React гайдлайны». Сборный файл-каталог получает размытые теги при индексации и не попадает в результаты на конкретный симптом
    - правильный паттерн: каждая stack-ловушка живёт в `fix-*` или `fact-*` **по теме симптома**, не по технологии. «Reconciliation trap уничтожает xterm при view-switch» → `fix-performance-lags.md` (там уже про unmount). «Portal click-outside сбрасывает state» → `сценарии-багов-design.md` секция Click-outside. «useCallback stale closure в Timeline auto-refresh» → `fix-incremental-reader-race.md`. Один шрам = одна секция в правильном тематическом файле
    - **тест перед записью**: «по какому симптому юзер откроет эту запись через год?» Если симптом про конкретную фичу (Timeline, Portal, Drag) → файл фичи. Если симптом универсальный («любая кнопка в Portal не реагирует») → `сценарии-багов-design.md`. Если симптом «React general» — это generic теория, которая выводится из MDN, **не записываем** по правилу AI-to-AI
    - generic стек-теория без симптома реального шрама (Loop closure trap с `var`, Init lock в Strict Mode, subscribe vs useEffect для batched transitions) — **не документируется**. AI знает это из обучения, запись не приносит новой информации

**калибровка по информации, не по шаблону** длина любой записи (methodology-секция, knowledge-факт, fix-абзац, anti-pattern в CLAUDE.md, inline-комментарий, описание PR) определяется плотностью невидимого контекста, не таргетом по строкам и не длиной примеров в правилах
    - тест на запись: *«через год при похожей задаче — даст ли запись ответ на „мой случай такой же?“ и „что делать?“ без растёкшихся в стороны слов?»*. Нет хотя бы по одному — добавь контекст. Да по обоим — лишние строки стрипай
    - тест на каждую строку: *«удалить → ответ на эти два вопроса всё ещё ясен?»* Да → строка лишняя, это padding для веса
    - принцип-однострочник пишется в строку. Принцип, прошедший три итерации с разными failure-modes, пишется длиннее — **но только потому что failure-modes сами по себе невидимый контекст**, не потому что «у методологии-секции должно быть N строк». Сначала набирается реальный материал → его форма даёт длину; нельзя выбрать длину сначала и потом подгонять материал
    - **антипаттерн — копировать длину соседей**: «у других methodology-секций по 25 строк, у моей пусть тоже». Соседи могут быть переписаны короче без потери — это норма, не баг. Консистентность по объёму ≠ консистентность по информации
    - **антипаттерн — копировать длину примеров в правилах/промпте**: примеры в `docs-rules.prompt.md` иллюстрируют *что бывает в записи* (типы блоков, структура), не targeting *сколько строк* должна занимать запись. Пример на 20 строк не означает «пиши 20 строк» — означает «вот один из вариантов того что внутри бывает»

**methodology-секция = проза, не шаблон.** Юзер диктует мысли голосом — они приходят как **связный поток**: «принцип такой-то, потому что [объяснение], а ещё юзер раз сказал [цитата], в проекте это вот в таком файле». Это **одна ткань**, не набор слотов. Записывается как сплошной текст в 1-3 абзаца, развёртывающий мысль естественно.
    - **никаких обязательных подзаголовков**: `**Симптом:**` / `**Реализация:**` / `**Тест:**` / `**В custom-terminal:**` / `**Антипаттерн:**` / `**Что не работает:**` / `**Когда применять:**` — НЕ являются скелетом методологической записи. Это не «структура для заполнения», это случайные маркеры которые **могут** появиться если в материале реально есть отдельный смысловой блок (3+ failure-modes за сессию, явный антипаттерн который юзер назвал словами, конкретный тест который юзер описал). По умолчанию — нет ни одного из них
    - **признак нарушения**: каждая methodology-секция выглядит как соседи (одинаковый набор bold-блоков). Это не консистентность, это template padding — Claude скопировал форму у первой секции которая случайно так была написана, и теперь все следующие воспроизводят ту же форму вне зависимости от материала
    - **признак нарушения 2**: блок `**Тест:**` или `**Антипаттерн:**` содержит выдуманный тест/антипаттерн, который **в сессии не звучал**. Юзер не описывал «прочитай вслух представив дизайнера другого проекта» — это Claude придумал «для полноты». Любой блок без origin'а в реальной сессии = padding
    - **привязка к проекту = inline-ссылка в потоке текста, не отдельный заголовок**. Вместо `**В custom-terminal:**` отдельной подсекцией → `… см. \`knowledge/fact-X.md::Section\`` в конце последнего абзаца. Сообщение «где этот принцип воплощён в коде» = одна ссылка, не самостоятельная глава
    - **идеальная форма methodology-секции:**
      ```
      ## Название принципа

      Сам принцип одной-двумя фразами в первом абзаце, с развёрткой контекста.
      Если есть прямая цитата юзера — inline в скобках или курсивом, не как
      отдельный «**Симптом:**» блок. Если есть конкретный антипример из живой
      сессии — следующим абзацем как продолжение мысли, не как «**Что не работает:**».

      В проекте: см. \`knowledge/fact-X.md\` (если применимо).
      ```
      Растёт **органически** по реальному материалу: 3 строки если принцип однозначный, 15 если действительно прошёл через несколько итераций с противоречивыми попытками. Без таргета и без шаблона
    - **self-check вопрос**: «для каждого bold-блока в этой секции — где в сессии прозвучал материал именно под этот блок? Если не назову конкретное место/цитату — удалить блок». Не «что убрал», а «что оставил и на каком конкретном основании»

**симптомы** при описании багов и проблем — обязательно записывать пользовательские симптомы
    - как техническая проблема выглядит для пользователя ("кнопка не нажимается", "данные пустые")
    - критично для семантического поиска: роутер ищет по симптомам, а не по имени файла

**заголовок раздела `###` имеет короткую преамбулу в курсиве**, если внутри раздела 3+ якоря-подтемы. Преамбула — 1-2 предложения через `*курсив*` сразу под `###` без bold, без отступа, без жирного якоря. Формулирует общий принцип всего раздела, чтобы читатель сразу понимал какой класс UI/багов/механики обсуждается ниже
    - *хорошо:* `### Модальные окна и панели` → `*Любое всплывающее окно — content-driven по геометрии. Высота функция числа элементов, ширина измерение от триггера, позиция через getBoundingClientRect. Конкретные сценарии и ловушки — секции ниже.*` → дальше 6 жирных якорей с конкретикой
    - преамбула снимает с каждого отдельного якоря необходимость повторять общую логику. Якорь становится про **частный случай** (resize модалки, Cmd-hover popover, click-popover для action), а не про повторное обоснование «почему вообще модалки так делают»
    - длина якорей внутри раздела может быть разной (от одной строки до трёх абзацев) — определяется плотностью невидимого контекста в конкретном правиле, не симметрией с соседями. Преамбула это не лимит длины, это контекст-настройка читателя
    - **не дублировать преамбулу в самих якорях**: общий принцип уже сказан сверху, в каждом якоре только специфика этого случая
    - **необязательна для раздела с 1-2 якорями**: там общий принцип читается из самого якоря, отдельная преамбула — шум. Преамбула оправдана когда раздел разросся до 3+ под-тем
    - применимо ко всем `###`-заголовкам в methodology, в `_intro.md` карте папок, в больших knowledge-файлах. Не применимо к `##` в knowledge-fact файлах (там сам файл = одна тема, преамбула в файловой шапке)

**UX-философия дублируется между `fact-*` и `methodology/design.md` — это ожидаемо, не дедуплицируем.** Распространённая ошибка — пытаться разделить «факт фичи» и «философию фичи» по разным файлам с cross-ссылкой. На практике это ломает основной use case: Claude при разработке новой фичи зовёт `docs_search` по теме фичи, Haiku-роутер тянет `fact-фичи.md` — а **не** тянет `methodology/design.md`, потому что симптом «реализую фичу X» далёк от симптомов в методологии (сформулированных как UX-принципы общего уровня). В результате Claude видит только сухое описание поведения, не видит «почему мы решили так а не иначе» и переделывает интуитивно — повторяя баги, которые уже один раз были разобраны
    - **правило**: если в fact-файле фичи есть UX-решение которое прошло через обсуждение/итерации/обоснование — записывай философию **тут же**, в том же fact-файле, отдельной секцией внутри (типа `## Почему именно так / Design rationale`). Не выноси отдельно в methodology «чтобы не дублировать»
    - параллельно тот же принцип в обобщённом виде идёт в `methodology/design.md` — без проектных имён компонентов, как переносимое правило. Это не дубль: fact содержит **конкретное применение в этой фиче**, methodology содержит **переносимую формулировку для других фич/проектов**
    - **сценарий активации обоих**: Claude разрабатывает новую фичу → `docs_search "X feature implementation"` → роутер тянет `fact-X.md` (там философия про X внутри) → Claude видит и поведение, и обоснование. Параллельно Claude разрабатывает другую фичу с похожим UX-классом → роутер тянет `methodology/design.md` (там переносимый принцип) → Claude применяет к новой фиче
    - **антипаттерн**: «вынесли философию из fact-X.md в methodology чтобы не повторять» → Claude разрабатывает X не зная философии, повторяет уже один раз решённую ошибку. Семантический поиск проектируется так что **одно знание может жить в двух местах**, потому что у двух мест разные точки активации
    - **граница с настоящим дублированием**: дубль = две полные копии одного и того же текста, которые расходятся со временем. Здесь — два *вида* одного знания: конкретное (в fact с именами компонентов) и обобщённое (в methodology без имён). Оба нужны, оба активируются по разным запросам

**активация документации = четыре точки входа, разная индексация**
    - **`docs_search` MCP (автоматически)** индексирует `docs/knowledge/`, `docs/methodology/`, `docs/product/`, `docs/meta/` через `.semantic-index.json`. Haiku-роутер выбирает 2-4 файла на запрос Claude. Это основной канал для разработки фич, отладки багов, продуктовых вопросов
    - **`CLAUDE.md` проекта (всегда)** грузится на каждый промпт. Сюда — крупные anti-patterns, обзор стека, философские принципы уровня всего проекта (например *«в этом проекте интерфейс для фермеров → каждая фича должна проходить тест „поймёт ли неопытный пользователь без подсказок?“»*). Не сюда — конкретные UI-паттерны, шрамы фич, design-философия отдельных модулей
    - **`~/.claude/CLAUDE.md` (всегда, глобально)** — правила общения юзер↔Claude (тон, формат, причинно-следственная цепочка). Через GT Editor deploy инжектится в каждый проектный `CLAUDE.md`. Не дублируется по проектам вручную
    - **`~/Global-Templates/🧩 Code-Patterns/docs-projects/` (глобально, в выдаче как `docs-global/*`)** — кросс-проектные паттерны (ux-паттерны/tooltip-resize-modal, инспектор-компонентов). **Индексируется глобально**: knowledge-server подмешивает её ручной `.semantic-index.json` в выдачу `docs_search` каждого проекта — записи приходят с префиксом `docs-global/`, fullPath ведёт в Global-Templates. Индекс ведётся руками: добавил файл → добавь entry (path + symptoms) в `docs-projects/.semantic-index.json`; проектный `docs_reindex` эту папку НЕ пересканирует. Связь с проектным `methodology/переносимый-дизайн.md` — направление миграции (созревшие принципы переносятся сюда), не runtime-зависимость
    - **права на `docs-projects/` разделены**: ОБНОВЛЯТЬ существующий глобальный файл, когда в сессии правилась реализующая его фича — обязанность промпта обновления документации (P6 в `docs-rules.prompt.md`). ДОБАВЛЯТЬ новые глобальные файлы/паттерны — только ручной шаг юзера с явным указанием (*«перенеси паттерн X в docs-projects»*); автоматически промпт этого не делает, даже если паттерн выглядит созревшим

**если Claude систематически не подтягивает класс файлов через `docs_search` — расширяй промпт роутера, не плоди новые типы файлов** распространённый антипаттерн при росте knowledge-базы: «Claude не находит дизайн-философию при реализации фичи → создадим новый тип файлов `flow-fix.md` / `ui-fix.md` / отдельную папку „глобальный дизайн“». Это не работает: новые типы файлов сами получают размытые теги при индексации и тоже не находятся
    - **правильное решение**: расширить промпт `knowledge-server.mjs` (или его роутер-инструкцию для Haiku) явным правилом активации. *«Когда запрос про реализацию новой UI-фичи — обязательно подтяни `methodology/design.md` и соседние `fix-*` по UI-теме. Когда запрос про исправление бага — `fix-*` по симптому + `fact-*` той же подсистемы. Когда запрос про маркетинг/юридику — только `product/`, knowledge не нужен.»*
    - **признак что нужен ремонт промпта, а не структуры**: юзер замечает что после реализации фичи в пост-промпте *«проверь knowledge ещё раз, не сломали ли мы что-то»* Claude находит важные файлы которые не подтянулись на этапе разработки. Это сигнал «роутер не активирует methodology/fix при запросах про реализацию», ремонт — одна строка в промпте сервера
    - **антипаттерн**: создание сборных файлов-каталогов («все принципы UX в одном месте», «все fix-ы по дизайну собраны через regex») — те же размытые теги, то же не находится
